Navigator usage and advanced setup

This navigator provides a static GitHub Pages site that lists repositories and links to their Pages.

Contents
- How the site works
- Enabling private repo support via Actions
- Deploying a Cloudflare Worker proxy (example)
- Deploying a simple Node proxy (example)

1. How it works
- The frontend prefers a static JSON at navigator/data/repos.json (generated by the workflow) to avoid GitHub API rate limits.
- If the JSON is absent the frontend falls back to calling the GitHub REST API directly (public data only).
- A GitHub Actions workflow (.github/workflows/generate-repos.yml) can generate navigator/data/repos.json daily or on push.

2. Enable private repo support
- Add a repository secret named REPO_LIST_PAT containing a personal access token (repo scope) if you want the workflow to list private repos.
- Edit .github/workflows/generate-repos.yml: set env INCLUDE_PRIVATE: true to enable the private artifact generation step.
- The workflow will NOT commit private data into the repo. Instead, it will upload a workflow artifact called repos-private-json. Download the artifact from the workflow run if needed.

Security notes
- Never put PATs into repository files. Use GitHub Secrets.
- If you need the navigator to show private data on the public Pages site, instead deploy a secure proxy (below) that only trusted users can access.

3. Cloudflare Worker proxy (recommended for small teams)
- Use a Worker to act as a proxy to the GitHub API using a secret token stored in Worker secrets. The Worker keeps the PAT secret and can implement caching and access control.

Deploy steps (summary):
1. Install Wrangler (Cloudflare CLI) and login: https://developers.cloudflare.com/workers/cli-wrangler/
2. Create a Worker and set a secret named GITHUB_TOKEN with your PAT: wrangler secret put GITHUB_TOKEN
3. Deploy worker code (see navigator/cloudflare-worker/index.js)
4. Point the navigator frontend to your worker endpoint (update ORG or proxy URL if needed).

Worker behavior: accepts GET /repos?org=ORG and returns JSON list of repos for ORG. Use caching headers and rate-limit checks.

4. Node proxy example (for Vercel/Heroku)
- A tiny Express app is included under navigator/proxy/node-server.js. Deploy this to any serverless host and set GITHUB_TOKEN env.
- The app exposes GET /repos?org=ORG and returns JSON.

5. Using the proxy from the frontend
- The navigator includes a data source selector (UI) that lets you choose: automatic (prefer static JSON), Cloudflare Worker, custom proxy URL, or direct GitHub API.
- When using a proxy that requires authentication, enter the API key in the "代理 API Key" field and save — the frontend will send it in the x-api-key header.

6. Next steps
- Deploy the Cloudflare Worker (see navigator/cloudflare-worker/README.md) and set GITHUB_TOKEN as a Worker secret.
- Deploy the Node proxy to Vercel/Render and set GITHUB_TOKEN and PROXY_API_KEY as environment variables.
- Consider adding server-side authentication and rotating keys using your hosting provider's secrets management.

如果需要，我可以为你部署 Cloudflare Worker 和 Vercel 代理（需要账号凭证）。或者，你也可以按照下面步骤手动部署。下面是自动部署（使用 GitHub Actions）与手动部署的说明：

自动部署（推荐）
- Cloudflare Worker：在仓库 Secrets 中添加 CF_API_TOKEN、CF_ACCOUNT_ID。然后在 Actions → Workflows 中运行“Deploy Cloudflare Worker”。工作流将使用 navigator/cloudflare-worker/wrangler.toml 执行 wrangler publish。
- Vercel Node 代理：在仓库 Secrets 中添加 VERCEL_TOKEN、VERCEL_ORG_ID、VERCEL_PROJECT_ID，然后运行“Deploy Node Proxy to Vercel”工作流。

手动部署（Cloudflare Worker）
1. 安装 wrangler：npm i -g wrangler
2. 在 navigator/cloudflare-worker/wrangler.toml 中设置 account_id
3. 在本地运行：wrangler secret put GITHUB_TOKEN（输入你的 PAT）
4. 部署：wrangler publish --env production

手动部署（Vercel）
1. 登录 Vercel，创建项目并选择导入 GitHub 仓库的 navigator/proxy 目录
2. 在项目设置中添加环境变量：GITHUB_TOKEN、PROXY_API_KEY
3. 部署并获取部署 URL（例如：https://navigator-proxy.vercel.app）

安全建议
- 始终把 Token/Key 放入平台 Secret/Environment，不要直接写入代码或仓库。
